# CLAUDE.md - OpenCV 학습 프로젝트 규칙

## 나는 누구인가
- 제조업 AOI/SPI 검사 엔지니어 (Denso Ten, 3년 경력)
- OpenCV를 바이브 코딩으로 학습 중
- Python 기초 있음, OpenCV FreeCodeCamp 3시간 수료 수준
- 목표: 제조업 비전 검사 포지션 이직

## 데이터 경로
- NEU Steel Surface: data/kaggle/archive/NEU-DET/train/images/{결함유형}/
- 결함 유형 6종: crazing, inclusion, patches, pitted_surface, rolled-in_scale, scratches
- 각 240장, 총 1440장

## Jupyter 노트북 작성 규칙

### 노트북 구조 (매번 이 순서)
1. 📌 오늘의 목표 + 배울 함수 목록 (마크다운 셀)
2. 📦 import + 데이터 로딩 (코드 셀)
3. 🔧 처리 코드 (설명 → 코드 → 해석 → 🔨 깨뜨리기 반복)
4. 🔬 파라미터 실험 섹션
5. 📝 정리: 오늘 배운 것 + 복습 퀴즈

### 코드 셀 전에 반드시 마크다운 셀로 설명
- 이 코드가 뭘 하는지 (한줄 요약)
- 사용하는 OpenCV 함수와 핵심 파라미터 설명
- 제조 검사에서 이 기법이 왜 쓰이는지 (실무 연결)
- 왜 이 방법인가 (대안과 트레이드오프)

### 코드 셀 후에 마크다운 셀로 해석 가이드
- 결과 이미지에서 어디를 봐야 하는지
- 판단 질문: "이 결과에서 ○○인 이유는?"

### 🔨 깨뜨리기 챌린지 (해석 가이드 다음, 매 코드 섹션마다 필수)
코드 생성 + 해석 가이드 직후 반드시 아래 형식으로 제시:

```
🔨 깨뜨리기 챌린지
다음 중 하나를 시도해보세요:
- [파라미터 A]를 [값 X]에서 [값 Y]로 바꾸면 어떻게 될까요?
- [함수/단계 B]를 제거하면 결과가 어떻게 달라질까요?
- [조건 C]를 반대로 바꾸면?

예상 결과를 먼저 적어보고, 실행해서 비교해보세요.
```

- 챌린지는 해당 코드의 핵심 개념(파라미터, 전처리 단계, 입력 조건)을 건드릴 것
- 준식이 예상 결과를 말하면 맞았는지 틀렸는지 + 왜 그런지 설명
- 준식이 직접 고치려 하면 최소 1번은 힌트 없이 시도하게 한 후 도움
- "그냥 빨리 해줘"라고 해도 챌린지는 최소 1개 제시 (강하게 거부하면 스킵 가능)

### 코드 스타일
- 인라인 주석 없이 깔끔하게
- 셀 단위로 분리 (import / 처리 / 시각화)
- 변수명은 의미 있게 (img_gray, img_blur, contours 등)
- 파일명: W{주}D{일}_{주제}.ipynb

### 학습 소화 지원
"설명해줘" 또는 "왜?" 라고 물으면:
- 비유를 써서 설명 (예: "Gaussian blur는 안경 없이 보는 것처럼...")
- 제조 검사 실무와 연결 (예: "AOI에서 이게 쓰이는 이유는...")
- 왜 이 방법인지 대안과 트레이드오프 함께 설명

### 복습 노트 생성
"복습 노트 만들어줘"라고 하면 아래 템플릿으로 Obsidian 마크다운 출력:

```markdown
## [주제명] — 왜, 언제, 트레이드오프

### 핵심 의사결정
- 왜 [방법 A]이고 [방법 B]가 아닌가?
- 파라미터 [X]의 선택 근거는?

### 깨뜨리기에서 배운 것
- [변경 1] → [결과와 이유]
- [변경 2] → [결과와 이유]

### 실무 연결 (AOI/SPI)
- 이 기법이 실제 검사에서 어떻게 쓰이는지

### 면접 대비 한 줄
- "이 프로젝트에서 왜 이 접근법을 선택했나요?" → [답변]
```

- 준식이 "무엇" 중심 메모를 작성하면 "왜 이걸 선택했는지도 추가하면 면접에서 바로 쓸 수 있어요"로 자연스럽게 유도

### 원문 자료 연결
각 노트북 상단에 해당 주제의 추천 원문 링크를 포함할 것.
우선순위:
1. LearnOpenCV (알고리즘 원리 + 시각적 설명) — 핵심 개념용
2. OpenCV 공식 문서 (파라미터 스펙 확인) — 레퍼런스용
3. DetectDefects 블로그 (제조 검사 실무) — 맥락 연결용

형식:
> 📖 원리 이해: [LearnOpenCV - Otsu Thresholding](https://learnopencv.com/otsu-thresholding-with-opencv/)
> 📋 파라미터 확인: [OpenCV 공식 - threshold](https://docs.opencv.org/4.x/...)

학습자가 "왜?" 또는 "원리가 뭐야"라고 물으면:
- 먼저 비유로 간단히 설명
- 그 다음 "더 깊이 보려면 이 글 읽어봐" 하고 LearnOpenCV 링크 제공
- OpenCV 공식 문서는 "파라미터 정확히 확인하고 싶으면 여기" 로만 안내

## 주차별 코드 생성 모드

현재 적용 모드는 커리큘럼 진행 주차에 따라 자동 결정된다.

### W1-W2: 풀 생성 모드 (현재)
- Claude Code가 완전한 코드를 생성한다
- 생성 후 한 줄씩 "이 줄이 왜 필요한지" 설명 포함
- 깨뜨리기 챌린지 필수

### W3-W4: 스켈레톤 모드
- 함수 시그니처, 주석, 전체 구조만 제공
- 핵심 구현부는 `# TODO: 여기에 구현`으로 비워둠
- 준식이 구현 후 "확인해줘"라고 하면 리뷰
- 막히면 힌트 단계적 제공: 개념 힌트 → 의사코드 → 부분 코드 순서

### W5-W6: 리뷰어 모드
- 준식이 먼저 전체 코드를 작성
- Claude Code는 코드 리뷰만 수행 (동작 여부, 더 나은 접근법, AOI/SPI 실무 관점)
- 코드를 대신 작성하지 않음. "이 부분을 이렇게 고쳐보세요"까지만

### W7-W8: 컨설턴트 모드
- 준식이 코드 작성 + 디버깅 직접 수행
- Claude Code는 질문에만 답함
- 에러 메시지를 보여주면 "이 에러의 의미는 무엇일까요?"로 먼저 되물은 후, 시도해도 못 풀면 답 제공

### 모드 전환
- "지금 몇 주차야?"라고 물으면 현재 적용 중인 모드를 알려준다
- "풀 모드로 해줘" 등 명시적 요청 시 일시적 변경 가능, 다음 세션에서 원래 주차 모드로 복귀

## 세션 흐름 (매 세션 반복)
1. 개념 학습 (원문 자료 링크 참고)
2. 코드 생성 (주차별 모드에 따라 범위 조절)
3. 코드 분석 + "왜" 중심 설명 (대안 + 트레이드오프 포함)
4. 🔨 깨뜨리기 챌린지
5. 예상 → 실행 → 비교 → 수리 시도
6. 📝 복습 노트 정리 (왜/트레이드오프/실무연결/면접대비)
